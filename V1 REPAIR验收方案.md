# V1 REPAIR 验收方案

生成时间：2026-04-20
对应文档：V1 BUG.md

这份文档用于验证 V1 BUG 清单中的问题是否已经被真正修复。

目标不是“代码改过了”，而是确认：
- 核心链路能稳定运行
- 前后端数据契约一致
- 搜索、上传、采集、图谱这些关键功能在真实场景下可用
- 修复没有引入新的回归问题

---

## 1. 验收目标

本次验收覆盖以下四类结果：

1. **运行稳定性**
   采集、上传、搜索、详情页、图谱等核心功能不再因模型字段不一致或链路错误而失败。

2. **数据正确性**
   分类、标签、结构化字段、关系边、搜索索引都能正确入库并正确返回。

3. **前后端一致性**
   前端页面使用到的字段，后端都能稳定提供；后端返回的数据结构前端能直接消费。

4. **回归安全性**
   修复后，现有测试通过；关键新增测试可覆盖本次修复点。

---

## 2. 验收范围

### 本次必须验收的 P0 / P1 问题

1. 视频上传链路写入不存在字段
2. 爬虫入库链路写入不存在字段
3. 关系分析链路字段名不一致、图谱边写入失败
4. 分类名称没有映射为数据库外键 ID
5. 结构化字段未统一 JSON 序列化
6. 全文搜索依赖的 `articles_fts` 不存在
7. 文章接口返回字段与前端实际需求不一致
8. 图谱节点跳转路由错误
9. 分类树没有数据来源
10. 修正接口把字符串直接写入分类外键

### 本次建议一并验收的 P2 风险项

1. Skill 反馈评分语义是否统一
2. API response schema 是否明确，是否还存在 ORM 裸返回

### 3.3 推荐验收环境

建议使用两套数据环境：

1. **干净环境**
   用于验证迁移、初始化、首次采集、首次上传。

2. **带历史数据环境**
   用于验证关系分析、搜索、分类筛选、图谱展示。

### 3.4 验收前置数据

建议准备：
- 一个文本文件：`sample.txt`

- 一个小音频文件：`sample.mp3`
- 至少 1 个可采集的数据源
- 至少 2 个分类数据
- 至少 2 篇可以建立关系的文章

## 4. 验收原则

每个修复项必须同时满足以下四点：

1. **功能通过**
   手工操作或 API 调用能够跑通。

2. **数据正确**
   数据库存储结果和 API 返回结果都符合预期。

3. **页面正确**
   前端页面能正常显示，不出现空字段、错误跳转或格式错乱。

4. **测试覆盖**
   对应后端测试已补充并通过。

只要其中一项不满足，该问题视为**未通过验收**。

---

## 5. 验收执行顺序

建议按以下顺序执行，避免前一个问题未修导致后一个问题无法验收：

1. 数据库迁移与初始化
2. 后端入库链路
3. 搜索链路
4. API 数据契约
5. 前端页面交互
6. 回归测试

## 6. 详细验收用例

## A-01 视频上传链路验收

### 对应问题
- 视频上传链路写入不存在字段
- 结构化字段未统一 JSON 序列化

### 验收目标
确认上传媒体文件后：
- 不会因 `Article.tags` 不存在而报错
- 会正确创建 `UploadTask`
- 会正确创建 `Article`
- 标签通过关联表保存
- `key_points`、`entities`、`segments` 存储格式正确

### 执行步骤
1. 启动后端服务
2. 在上传页上传 `sample.mp3`
3. 等待任务处理完成
4. 查看返回结果和数据库记录

### 预期结果
- 上传接口返回 200
- `upload_tasks.status = completed`
- 成功生成 `article_id`
- `articles` 表中有新记录
- `article_tags` 表中存在对应关联数据
- `video_transcripts.segments` 为合法 JSON 字符串或经过统一 schema 处理后的结构
- 前端上传列表能正常展示任务状态

### 判定失败的信号
- 报 `NameError: Article is not defined`
- 报 “invalid keyword argument: tags”
- 文章创建成功但没有标签关联
- 前端展示任务成功，但数据库文章未落地

### 验收证据
- 上传接口响应截图或日志
- `upload_tasks`、`articles`、`article_tags`、`video_transcripts` 查询结果

---

## A-02 爬虫入库链路验收

### 对应问题
- 爬虫入库链路写入不存在字段
- 分类结果未写入外键 ID
- JSON 字段未序列化

### 验收目标
确认手动触发采集后：
- 不会因 `tags` 字段不存在而失败
- 文章可成功入库
- 分类被正确写入 `primary_category_id` / `secondary_category_id`
- 标签正确写入关联表
- `key_points`、`entities` 存储格式正确

### 执行步骤
1. 新建一个数据源
2. 调用“立即采集”
3. 查看新增文章
4. 检查数据库

### 预期结果
- `/api/crawler/trigger/{id}` 返回 200
- 返回 `article_ids` 非空
- `articles` 表成功新增记录
- 分类 ID 非空且对应 `categories.id`
- `article_tags` 存在对应数据
- `key_points`、`entities` 可被 API 正确解析后返回数组

### 判定失败的信号
- 触发采集时报模型字段错误
- 文章入库成功但分类为空
- 文章入库成功但标签丢失
- 结构化字段返回字符串脏格式

### 验收证据
- 触发采集响应
- `articles`、`article_tags`、`categories` 查询结果

## A-03 关系分析和知识图谱边验收

### 对应问题

- `KnowledgeEdge` 写入失败

### 验收目标
确认关系分析结果能真正落入 `knowledge_edges` 表，并能在图谱接口中查到。

3. 检查 `knowledge_edges`
4. 请求图谱接口

### 预期结果
- `knowledge_edges` 表新增记录
- `source_article_id`、`target_article_id` 正确
- 图谱接口 `/api/graph/articles/{id}` 返回 links 非空

### 判定失败的信号
- 后处理阶段报 `KnowledgeEdge` 未定义
- 图谱节点有，关系线没有
- 图谱接口返回空 links，但数据库理论上应有关系

### 验收证据
- `knowledge_edges` 查询结果
- 图谱接口响应 JSON
- 图谱页面截图

## A-04 分类映射验收

### 对应问题
- AI 分类结果没有真正写回数据库外键

### 验收目标
确认 AI 返回的是分类名称时，系统能正确映射到 Category ID。

### 执行步骤
1. 预先插入分类树：如“宏观经济 / 货币政策”
2. 处理一篇明显属于该分类的文章
3. 查看文章记录和接口返回

### 预期结果
- `articles.primary_category_id` 有值
- `articles.secondary_category_id` 有值
- 这两个值能在 `categories` 中查到对应名称
- 文章详情接口返回分类名称而不是裸 ID

### 判定失败的信号
- 数据库只保存了字符串名称
- ID 为空或找不到对应分类
- 页面显示 `undefined` 或显示数字 ID

### 验收证据
- `articles` 与 `categories` 联查结果
- 文章详情接口 JSON

## A-05 JSON 结构化字段验收

### 对应问题
- `key_points`、`entities`、`segments` 未统一序列化

### 验收目标
确认所有结构化字段在数据库和 API 层都保持一致的格式。

### 执行步骤
1. 生成一篇含摘要、实体、要点的文章
2. 上传一段音频生成 transcript
3. 查看数据库原始值
4. 查看 API 返回值

### 预期结果
- 数据库存储为合法 JSON 字符串，或使用统一 schema 输出
- API 返回给前端时是数组 / 对象，不是 Python 字符串样式
- 前端页面可直接渲染，不再需要脏判断

### 判定失败的信号
- 页面出现 `['a', 'b']` 这种字符串展示
- 前端需要手工 `JSON.parse` 才能勉强显示
- 同一字段在不同接口中格式不一致

### 验收证据
- 数据库字段原始值
- 文章详情接口响应
- 页面展示截图

## A-06 全文搜索与 FTS5 验收

### 对应问题
- `articles_fts` 表不存在

### 验收目标
确认全文搜索不再依赖异常 fallback，而是真正通过 FTS5 工作。

### 执行步骤
1. 执行数据库迁移
2. 检查 `articles_fts` 是否存在
3. 插入一篇含关键词的文章
4. 调用 `/api/search?q=关键词`
5. 更新文章标题或内容，再次查询
6. 删除文章，再次查询

### 预期结果
- 数据库中存在 `articles_fts`
- 插入文章后 FTS 可命中
- 更新文章后 FTS 同步更新
- 删除文章后 FTS 中对应记录消失

### 判定失败的信号
- 搜索接口只能靠 fallback 命中
- 更新/删除文章后搜索结果不同步
- 迁移完成后依然找不到 `articles_fts`

### 验收证据
- `sqlite_master` 查询结果
- 搜索接口结果对比

### 对应问题

### 预期结果

- `id`

- `secondary_category`
- `tags`

- `entities`
- `created_at`

- 接口返回 ORM 里不存在前端需求字段
- 前端页面仍出现空分类、空标签
- 同一文章在列表页和详情页结构不一致

### 验收证据
- 列表接口响应
- 详情接口响应
- 页面截图

## 10. 建议验收顺序

推荐按这个顺序执行：

1. 先验数据库迁移和 FTS
2. 再验 crawler / upload 入库
3. 再验 article / search / graph API
4. 最后验前端页面和交互
5. 最后做回归测试和冒烟测试

这样可以避免前端问题掩盖后端真实问题。

---

## 11. 备注

当前项目的问题重点不是“有没有报红”，而是“真实链路能不能稳定跑通”。

因此本验收方案强调：
- 不只看代码改没改
- 不只看测试过没过
- 一定要看数据库结果、API 结果和页面结果是否一致

只有三者一致，才算真的修好了。
