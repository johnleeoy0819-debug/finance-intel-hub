# V2 BUG 清单、修复方案与验收方案

生成时间：2026-04-20

这份文档基于当前仓库代码的二次复审整理，只保留“当前代码里仍然存在”的问题。

说明：
- 本次 V2 不是重复 V1，而是针对当前代码状态重新核对后的结果。
- 已在当前代码中修复的问题，不再重复列为 V2 BUG。
- 本次重点仍然是运行时问题、数据一致性问题、页面崩溃风险和验收缺口，不讨论代码风格。

---

## 一、复审结论

当前项目已经明显优于 V1 阶段，之前最严重的一批问题已经被修掉，包括：

- FTS5 迁移已经存在，`articles_fts` 不再是“完全缺失”的状态
- 分类与标签接口已经补上，分类树已有后端数据来源
- 文章接口已经有较完整的响应组装，不再是纯 ORM 裸返回
- 图谱点击详情页的主路由已经对齐到 `/article/:id`
- 点赞 / 点踩反馈语义已经比 V1 时更一致

但当前代码仍然存在一批“主流程一执行就会暴露”的问题，集中在四条链路：

1. 文章采集 / URL 入库
2. 视频转录上传
3. Wiki 自动更新
4. 前端页面容错与数据一致性

优先级说明：
- P0：直接导致核心流程失败，或页面在异常数据下直接崩掉
- P1：功能表面能跑，但结果不可信、会产生脏数据，或运维上不可观测
- P2：当前不一定马上炸，但会持续积累为回归风险或维护风险

## 三、当前 BUG 清单

## P0 级 BUG

### 1. 文章采集和 URL 入库链路都会在回链更新时触发 `NameError`

影响文件：
- `server/src/core/crawler.py`
- `server/src/core/url_processor.py`

问题：
- 两个文件都调用了 `update_backlinks(db, ra.id, [article.id])`
- 但函数没有在文件内导入

影响：
- 采集来源文章时，后处理阶段会在更新 backlinks 时直接报错
- 手动提交 URL 文章时，也会在同一逻辑处失败
- 这不是“数据少一部分”，而是采集 / URL 入库主流程会被打断

修复方案：
1. 在 `crawler.py` 和 `url_processor.py` 中显式导入 `update_backlinks`
2. 修复后补最小回归测试，至少覆盖一次成功采集和一次 URL 入库

---

### 2. 视频上传链路在转录落库阶段会因 `json_dumps_field` 的导入顺序报错

影响文件：
- `server/src/core/video_processor.py`

问题：
- `process_video_upload()` 在前半段就调用了 `json_dumps_field(transcript["segments"])`
- 但 `json_dumps_field` 是在函数后半段才通过局部导入引入
- 在 Python 作用域规则下，这会让它被视为函数局部变量，从而在首次使用处触发未绑定错误

影响：
- 视频 / 音频上传并不是“可能不稳定”，而是转录记录保存阶段就可能直接失败
- 上传任务会进入失败状态，后续文章生成也无法继续

修复方案：
1. 把 `json_dumps_field` 提到文件顶层导入，或在函数顶部统一导入
2. 避免在同一函数里“先使用，后局部导入”
3. 补一个覆盖 `VideoTranscript.segments` 的回归测试

---

### 3. Wiki 更新路径使用 `datetime.utcnow()`，但模块未导入 `datetime`

影响文件：
- `server/src/core/wiki_compiler.py`

问题：
- `compile_topic()` 和 `compile_index()` 已经把旧的 `db.func.now()` 改成了 `datetime.utcnow()`
- 但模块顶部没有导入 `datetime`

影响：
- 新建 Wiki 页面时不一定暴露
- 但更新已有 Wiki 页面或索引页时会直接抛 `NameError`
- 这会打断 Wiki 重新编译和索引刷新流程

修复方案：
1. 在 `wiki_compiler.py` 顶部显式导入 `datetime`
2. 覆盖“已有 Wiki 页面再次编译”和“索引页刷新”两个场景

---

## P1 级 BUG

### 4. 搜索接口会静默吞掉 FTS 失败，真实问题被隐藏

问题：
- 全文搜索和混合搜索都用了宽泛的 `except Exception`
- 失败后直接 fallback，没有日志、没有告警

影响：
- 如果 FTS 表损坏、迁移没应用完整、查询语法异常，系统表面上仍会返回结果
- 但团队很难第一时间发现“全文搜索其实已经降级成简单 LIKE”

修复方案：
1. 保留 fallback，但必须记录 warning 级别日志
2. 最好把 FTS 不可用和普通无结果区分开
3. 增加异常分支测试，至少验证日志会被打出

---

### 5. 调度器试图隔离非法 cron，但告警分支自身会因 `logger` 未定义再次失败

影响文件：
- `server/src/core/scheduler.py`

问题：
- `add_crawl_job()` 已经加入了 `try/except` 来捕获非法 cron
- 但 `except` 里直接调用 `logger.warning(...)`
- 该模块没有定义 `logger`

影响：
- 当某条 Source 的 cron 非法时，本来应该“跳过坏配置并记日志”
- 现在会在异常处理分支再触发 `NameError`
- 结果是调度器对坏配置仍然不具备真正的隔离能力

修复方案：
1. 在 `scheduler.py` 中补齐 `logging` 导入和模块级 `logger`
2. 保留当前坏 cron 跳过策略
3. 增加一条非法 cron 的测试用例，确认不会影响其它任务注册

---

### 6. Chat 页面内部链接使用原生 `<a href>`，会触发整页刷新并丢失单页状态

影响文件：
- `web/src/pages/Chat.tsx`

问题：
- Chat 页面中跳转 Wiki 和文章详情时，使用的是原生 `<a href>`
- 这会绕过 React Router 的单页导航

影响：
- 点击引用来源或 Wiki 综述时会整页刷新
- 聊天上下文和当前页面内状态会被清空
- 不是数据错误，但会显著破坏对话体验

修复方案：
1. 改用 `react-router-dom` 的 `Link`
2. 统一内部路由跳转策略，避免页面重载
3. 补一个最小前端 smoke test，验证点击来源后仍保持 SPA 导航

---

### 7. Chat 的反馈和保存 Wiki 操作会静默吞掉失败，用户看不到任何失败提示

影响文件：
- `web/src/pages/Chat.tsx`

问题：
- `handleFeedback()` 和 `handleSaveToWiki()` 都用了空 `catch {}`
- 请求失败后页面既不提示错误，也不解释是否重试成功

影响：
- 用户会误以为反馈或保存已执行
- 这类问题不会直接崩页面，但会让交互结果不可感知

修复方案：
1. 至少在失败时展示提示信息
2. 保留当前按钮状态，不要假装成功
3. 为失败分支补一个前端行为测试

---

### 8. 当前测试覆盖仍不足以证明本轮修复不会回归

问题：
- 当前后端测试已经比 V1 时丰富很多，但仍缺少这几类关键回归覆盖：
  - Wiki 重编译和索引刷新路径里的 `datetime` 问题
  - 非法 cron 的隔离和日志路径
  - Chat 页内部导航是否保持单页路由
  - Chat 的反馈 / 保存失败提示

影响：
- 即使这轮问题修掉，只要后续再改一点代码，也很容易再次回归

修复方案：
1. 后端至少补齐核心接口和核心异常分支测试
2. 前端至少增加页面级 smoke test，覆盖最容易回归的交互路径

---

## P2 级 BUG / 风险项

### 9. 规则设置接口没有长度约束，用户规则可以无限放大 Prompt 负担

影响文件：
- `server/src/api/settings.py`

问题：
- `rules` 目前没有实际长度上限

影响：
- 用户规则如果过长，会抬高 prompt 成本，增加截断风险
- 这类问题不一定马上报错，但会逐步污染 AI 侧行为和成本

修复方案：
1. 在请求 schema 或接口层增加合理长度限制
2. 最好返回明确错误提示，而不是静默接受超长规则

---

### 10. 语义搜索分支会把全局文章列表清空，页面状态与展示状态出现分裂

影响文件：
- `web/src/pages/Library.tsx`

问题：
- 语义搜索成功后，`searchResults` 被正确设置
- 但同时又执行了 `setArticles([], items.length)`

影响：
- 当前页面虽然还能依赖 `searchResults` 渲染结果
- 但全局 store 中的 `articles` 被清空，后续分页、统计、页面切换时容易出现状态错乱

修复方案：
1. 统一三种搜索模式的 store 写入策略
2. 要么都只用局部 `searchResults`，要么都同步维护 store，不要一半一半

---

### 11. 前端 `Article.key_points` 类型声明比真实运行时更窄，属于数据契约漂移

影响文件：
- `web/src/types/index.ts`
- `web/src/pages/ArticleDetail.tsx`

问题：
- 类型定义里 `key_points?: string[]`
- 但页面代码实际按“可能是 JSON 字符串，也可能是数组”来处理

影响：
- 这说明前后端契约还没有完全稳定
- 类型系统无法完整表达真实输入，后续容易重复出现“临时 parse 一下”的补丁式代码

修复方案：
1. 后端明确统一响应格式，前端按最终格式收窄类型
2. 若过渡期必须兼容历史数据，建议在 API client 层统一清洗，而不是分散在页面组件里

## 四、建议修复顺序

建议按下面顺序处理，避免上一个问题没修完就卡住下一个问题：

### 第一阶段：先修主流程阻塞项

1. `crawler.py` / `url_processor.py` 缺失 `update_backlinks` 导入
2. `video_processor.py` 的 `json_dumps_field` 导入顺序
3. `wiki_compiler.py` 的 `datetime` 导入缺失

目标：
- 先确保采集、URL 入库、视频上传、Wiki 更新这几条核心链路真正能跑通

### 第二阶段：修数据一致性和可观测性

1. 搜索 fallback 记录日志
2. 调度器非法 cron 的告警路径补全
3. Chat 的失败提示和内部导航修正

目标：
- 让系统不只是“跑了”，而是结果可信、问题可查

### 第三阶段：修前端页面稳定性

1. Library 语义搜索 store 写入策略统一
2. Chat 反馈和保存 Wiki 的失败可见性
3. Chat 内部路由改为 SPA 导航

目标：
- 避免后端修好了，前端又因为脏数据或组件使用错误把页面打崩

### 第四阶段：补测试，锁住本轮修复成果

1. 后端补异常分支和一致性测试
2. 前端补最小 smoke test
3. 重新跑 `pytest`、TypeScript 检查和前端构建

## 五、验收方案

本次验收目标不是“代码改过了”，而是确认：

1. 主流程真的能跑通
2. 搜索、调度和 Wiki 更新等跨模块行为是可观测且一致的
3. Chat 页导航和反馈行为不会悄悄失败
4. 本轮修复有最小自动化回归保护

## A. 后端主流程验收

### A-01 文章采集 / URL 入库验收

对应问题：
- P0-1

执行步骤：
1. 启动后端
2. 新建一个可用数据源并触发采集
3. 再提交一个普通文章 URL
4. 检查接口响应和数据库

预期结果：
- 采集和 URL 入库都返回成功
- 不出现 `update_backlinks` 未定义错误
- 新文章可以成功落库
- 相关 backlinks 字段被正常更新

失败信号：
- 报 `NameError: update_backlinks is not defined`
- 文章已生成，但后处理阶段中断

---

### A-02 视频上传验收

对应问题：
- P0-2

执行步骤：
1. 上传一个小音频或视频文件
2. 等待任务进入转录和文章生成阶段
3. 检查 `upload_tasks`、`video_transcripts`、`articles`

预期结果：
- 上传任务最终为 `completed`
- `video_transcripts.segments` 被正确序列化保存
- 生成对应文章，不出现局部变量未绑定错误

失败信号：
- 报 `UnboundLocalError` 或与 `json_dumps_field` 相关的异常
- 转录已完成，但文章链路未完成

---

### A-03 Wiki 更新验收

对应问题：
- P0-3

执行步骤：
1. 准备一篇带 `entities` 的文章并触发 Wiki 级联更新
2. 对同一主题连续执行两次 Wiki 编译
3. 再执行一次索引页更新

预期结果：
- 不出现 `datetime` 未定义错误
- 首次编译成功
- 已有 Wiki 页面再次编译也成功

失败信号：
- `NameError: datetime is not defined`

- P1-5

- 非法 cron 被跳过并记录日志

失败信号：

## C. 前端稳定性验收

- P2-10

执行步骤：

## D. 自动化回归验收

本轮建议最少补下面这些自动化验证：

### 后端必须补

1. 采集 / URL 入库成功触发 backlinks 更新
2. 视频上传链路不会因 `json_dumps_field` 作用域问题失败
3. Wiki 页面重复编译不会因 `datetime` 导入缺失失败
4. FTS fallback 会记录日志
5. 非法 cron 不影响其它任务注册

### 前端建议补

1. Chat 内部文章 / Wiki 链接保持 SPA 跳转
2. Chat 反馈失败时给出提示
3. Chat 保存 Wiki 失败时给出提示

---

## 六、验收通过标准

只有同时满足下面四条，才算本轮 V2 修复通过：

1. 采集、URL 入库、视频上传、Wiki 更新四条主流程可以真实跑通
2. 搜索降级和调度器坏配置都具备可观测性且不会把系统拖垮
3. Chat 页内部导航、反馈和保存 Wiki 行为对用户是可感知且可用的
4. 至少补齐关键回归测试，并通过后端测试与前端构建验证

如果只是“代码改了”，但以上任一条仍不满足，就不能算 V2 完成。
